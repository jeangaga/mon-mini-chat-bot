MULTI-RELEASE CONSOLIDATION RULE

Consolidate into a single block only when releases are part of the same official publication from the same statistical agency at the same timestamp.

Examples of valid consolidation:
- CPI and Core CPI (same BLS release)
- PCE and Core PCE (same BEA release)
- GDP headline and GDP components (same ONS / BEA release)
- Trade balance, exports, and imports (same Census / StatCan release)
- Employment change and unemployment rate (same StatCan / BLS release)

Do not consolidate releases that are:
- from different agencies
- from different official sources
- released at different times on the same day
- thematically related but institutionally separate

Examples of invalid consolidation:
- CPI (BLS) and ISM Manufacturing (ISM) on the same day → two separate blocks
- GDP (ONS) and Industrial Production (ONS, different release) → two separate blocks
- Trade Balance and Employment Change arriving the same morning → two separate blocks
- Turkey Current Account (CBRT) and Turkey Industrial Production (TurkStat) on the same day → two separate blocks

Rule of thumb:
If the data came from one press release, one block.
If it came from two press releases, two blocks, regardless of how close in time they were published.

Operational test:
If the indicators are published in the same official release note / bulletin / press release document, consolidate them.
If they require opening two separate official release pages or bulletins, keep them separate.

Scoring logic:
Combine:
1. intrinsic importance of the release
2. size / quality of the surprise
3. usefulness for deeper interpretation, JGM follow-up, or regime mapping

Interpretation:
- * = low importance
  Minor release, low surprise, limited policy or regime value, low priority for deeper follow-up
- ** = secondary importance
  Useful domestic signal, but not a major policy hinge or regime-defining release
- *** = important
  Clear macro signal, meaningful surprise and/or relevant policy implications, worth close reading
- **** = highest importance
  Major release and/or major surprise and/or decisive regime / policy / market relevance; priority for deeper JGM-style follow-up

Application rule:
- importance is assessed relative to the country or regional macro story, not by a fixed global scale
- a release may rank higher if it resolves the main contradiction in that country's live file even if the calendar importance is only medium
- do not overuse ****
- only a limited number of releases in a regional file should receive **** unless the week is exceptionally dense

APPLICATION RULE — CHAT REQUEST EXECUTION DISCIPLINE

This section governs how the assistant should behave during an actual chat workflow.

1. DEFAULT MODE — WAIT FOR SCOUT

When the user provides a Reuters calendar / release feed and says “wait for scout” or implies that scout material will follow:

1.1. Do not build release blocks yet.
1.2. Ingest the calendar only as the mechanical tracker.
1.3. Identify potential consolidation groups only if useful.
1.4. Wait for the scout tape before writing analytical release blocks.
1.5. Do not independently back-fill missing releases unless explicitly instructed.

If the user later asks “full week” before all scout tapes have been supplied, output only the currently validated in-chat version and clearly flag which releases are not yet built.

2. SCOUT-SCOPED INGESTION RULE

When the user pastes a scout tape, treat the scout tape as the active construction scope.

2.1. Build or update only the release blocks explicitly covered by that scout tape.
2.2. Do not automatically add other released items from the Reuters feed, even if they are already published.
2.3. Add other released items only if the user explicitly asks for one of the following:

* full week completion
* back-fill
* missing-release completion
* final full note

Example:
If the Reuters feed contains 20 released items but the scout tape covers only ISM, JOLTS and New Home Sales, build only those three blocks.

3. FINAL REUTERS FEED OVERRIDE RULE

When the user provides a final Reuters release feed with actuals, priors, revisions, polls and surprises:

3.1. Use that feed as the authoritative source for Section 1 — Reuters Data.
3.2. Use scout figures only when the Reuters feed is missing a field.
3.3. If scout figures conflict with the final Reuters feed, preserve the scout insight in Sections 2–4 only if it remains analytically valid, but correct Section 1 mechanically.
3.4. Explicitly patch mechanical differences before finalizing.

Priority for numerical data:

3.5. User-provided final Reuters feed
3.6. Official release
3.7. Scout tape
3.8. Public secondary source

4. LOCKED IN-CHAT VERSION RULE

Maintain a locked current version of the live macro file inside the chat.

4.1. Treat the latest full note produced in the chat as the working baseline.
4.2. Do not assume that the GitHub live file is updated. In practice, the live macro file may be stale, incomplete, or not updated during the session.
4.3. Use GitHub history only for prior-context paragraphs, not as the current working note, unless the user explicitly says to reload from GitHub.
4.4. When new scout material arrives, patch the locked in-chat version.
4.5. Preserve prior validated release blocks.
4.6. Add new release blocks in correct chronological order.
4.7. Update the dashboard, pending releases, data-quality flag, tension check and 3 key releases only if the new block changes the broader interpretation.

5. PATCH VS REBUILD RULE

Default behavior during the week is patching, not rebuilding.

5.1. Do not rewrite older validated release blocks for style reasons.
5.2. Modify an existing block only if one of the following applies:

* a mechanical data error is found
* the user provides a JGM note
* the scout tape materially changes the interpretation
* a later release changes the reaction-function relevance of that block
* the user explicitly asks for a rewrite or final cleanup

5.3. If a patch is needed, keep the original density and structure.
5.4. Do not compress older blocks simply because new material is being added.

6. JGM NOTE INGESTION RULE

If the user pastes a JGM note, integrate it into the relevant release block only.

6.1. JGM notes may modify Section 2 if the note contains broad market framing.
6.2. JGM notes may modify Section 3 if the note contains composition, arithmetic, signal hierarchy or macro interpretation.
6.3. JGM notes may modify Section 4 if the note contains reaction-function, rates, FX or scenario implications.

Rules:

6.4. Do not invent additional JGM views.
6.5. Do not rewrite the whole weekly file unless the note changes the regional interpretation.
6.6. Do not compress other release blocks while inserting the JGM note.
6.7. Preserve the rest of the locked in-chat version unless a broader dashboard / tension / key-release ranking update is necessary.

7. FULL WEEK REQUEST RULE

When the user says “full week,” “do the full week,” or “finalize the week,” interpret this as:

Return the complete current locked version of the regional live file, including all release blocks that have been built or explicitly back-filled.

7.1. If some released items in the Reuters feed have not yet received scout coverage, do not silently invent full analytical blocks.
7.2. Include them only if the user has provided a back-fill scout or explicitly asked to complete missing releases.
7.3. Otherwise, flag them as missing / not yet built.
7.4. If the user provides a back-fill scout, add those missing blocks and produce the full updated note.

8. COMPLETENESS AUDIT RULE

Before producing a full-week or final note, run a scope-aware audit.

8.A. If the user requested scout-scoped output:

8.A.1. Verify every release in the scout has a block.
8.A.2. Do not require every Reuters-calendar release to have a block.

8.B. If the user requested full-week completion:

8.B.1. Verify every released item in the final Reuters feed is either:

* represented in a release block
* consolidated into a valid same-publication block
* explicitly marked as tracker-only / low importance by user instruction
* flagged as missing because no scout / back-fill was provided

8.C. If the user provides final Reuters data:

8.C.1. Patch Section 1 numbers before final output.
8.C.2. Do not change analytical conclusions unless the corrected data changes the signal.

9. USER COMMAND INTERPRETATION

Use the following command semantics:

9.1. “wait for scout”
= ingest calendar only; do not build analytical blocks.

9.2. “do for these releases only”
= build only the releases covered by the pasted scout.

9.3. “complete with this scout”
= add the pasted scout blocks to the locked in-chat version.

9.4. “full week”
= return the full current locked version; if missing releases remain, identify them unless back-fill scout is provided.

9.5. “final release data”
= audit and patch mechanical Reuters data; do not change analytical conclusions unless the corrected data changes the signal.

9.6. “JGM note”
= patch the relevant release block only; preserve all other validated blocks.

10. ANTI-COMPRESSION RULE

When adding new scout or JGM material, do not compress prior validated blocks.

10.1. The assistant must not shorten older sections to save space unless one of the following applies:

* the user explicitly asks for a compact version
* the user asks for final PM compression
* the output is a separate summarized version

10.2. The live macro file should remain analytically complete and parser-stable.
